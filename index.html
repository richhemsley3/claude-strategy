A strategy for transitioning a large, regulated design organization from Figma-centric to code-first practice — with governance, cross-functional alignment, and startup speed where it counts.

Three Simultaneous Transitions

This strategy navigates three interrelated shifts happening at the same time. Success requires addressing all three.

Practice Transition

From Figma-centric design to code-first design. The primary artifact becomes a working prototype — automatically testable for compliance, eliminating the design-to-engineering interpretation gap.

Operating Model Transition

From uniform process to a two-mode model. Scrappy LOBs move at startup speed. Enterprise LOBs maintain audit-ready discipline. The foundation layer is shared; the operating weight varies.

Cross-Functional Transition

From siloed handoffs to shared infrastructure. Design standards become ambient in the workspace. Every function that touches UI works from the same conventions.

The Speed Argument

Real gains are 2-3x for end-to-end feature delivery, concentrated in handoff and clarification. Not 10x.

Where Speed Gains Are Real

  • Interpretation gap eliminated: 3-7 days per feature. The 15 Slack threads after every Figma handoff are answered by the prototype itself.
  • Spec gap eliminated: 1-2 days per feature. The design-handoff skill follows a checklist, not memory.
  • Rapid variant testing: 2-3x more variants. Two working prototypes in a single Claude session, tested the same day.
  • Component reuse compounds. Every component starts from your design system's conventions automatically.

Where Speed Claims Are Overstated

  • "10x faster" is marketing. Real gains concentrate in handoff and clarification, not raw design production.
  • First-time setup is slower. Learning curve takes 2-4 weeks before net positive. Pilot with adaptable designers.
  • Speed only helps if production is the bottleneck. Audit the actual bottleneck per team before pitching Claude as the solution.

The Regulatory Argument

In a regulated industry, automated-first quality assurance is fundamentally stronger than manual-only review.

A code prototype can be automatically tested. A Figma mockup cannot.

Accessibility validation, token compliance, keyboard navigation, semantic HTML, component API correctness — all automatically checkable on code. Every check runs on every commit. Violations are caught before human review, not during it.

Eliminates Translation Drift

The gap between "what the designer intended" and "what the engineer built" is the largest source of quality loss. When the designer's artifact IS code, that gap collapses.

Creates an Audit Trail

Every production-bound component has documented evidence: which skills were active, what automated checks passed, which designer reviewed and approved, when.

The Practice Transition: Figma to Code-First

Code-first means designers use Claude Code to generate working prototypes as the primary design artifact. It does NOT mean designers write production code or that Figma disappears.

Critical Principle

Think in Figma (or on paper, or a whiteboard). Build in Claude. Review in whatever medium serves the conversation. Code-first must not become code-only.

Phase 1 Use Cases

  • Design critique: Screenshots of Figma work evaluated against principles and accessibility. Builds trust without changing workflow.
  • Spec generation: After completing design in Figma, design-handoff skill generates spec packages. Immediate value to engineering.
  • Hypothesis challenge: Forces problem validation before prototyping begins — what user problem, what evidence, what assumptions, what's the simplest testable version.
  • Sandbox prototyping: Willing designers generate components using the Foundation skill. Sandbox repo, not production.

The Two-Mode Operating Model

A shared quality floor with variable operating weight. The foundation is consistent — the speed varies by context.

The Transition Gate

Why Three Specific Questions Instead of "Review and Approve"

The human sign-off asks: 1) Does this work at mobile width? 2) Do error/edge states make sense? 3) Does it meet disclosure/regulatory requirements? Specific questions get real answers. "Review and approve" gets rubber stamps.

Product Management Collaboration

Five specific ways the PM-design dynamic changes with code-first workflows — and how to navigate each.

PRD Quality Becomes Instantly Visible

A vague PRD produces a vague prototype in minutes — making ambiguity instantly visible. The read-filter-prd skill extracts UI/UX requirements into a structured brief. Gaps surface in minutes instead of days.

PMs See Working Prototypes Earlier

PMs interact with working prototypes within the first few days. Feedback becomes "I tried this on mobile and it didn't work" instead of "I think this might be confusing."

Requirements Traceability

Trace from PRD requirement to design decision to prototype to production. Every decision in the chain is documented and auditable.

Acceptance Criteria Validation

PMs validate AC against working prototypes immediately instead of weeks later in QA. Catches "design doesn't do what we agreed" before engineering begins.

PMF Team Workflow: Hypothesis to Testable Prototype in 1-2 Days

PM drafts hypothesis → PM and designer run challenge-hypothesis together → refine in real-time → designer generates prototype (30-60 min) → PM validates against AC → team tests with users → synthesize → next hypothesis.

Boundary: PMs Should Not Generate UI Prototypes

The design org maintains ownership of design decisions. If PMs generate their own prototypes, you get two versions of the truth and a political problem.

Engineering Alignment

If design outputs code prototypes but engineering doesn't use the same Foundation skill, you've moved the drift gap rather than eliminating it.

The Unified Foundation Skill

The Foundation skill is not a "design skill." It's a UI standards skill for everyone who generates UI code. When an engineer asks Claude to "add a filter dropdown," the Foundation skill loads and the component uses your tokens, conventions, and accessibility requirements. One design system, both disciplines.

Governance Model

Hybrid with a strong center. Purely federated governance has worked at exactly zero organizations.

Core Principle

LOB skills can add requirements. They cannot remove or contradict foundation requirements. The hierarchy mirrors token architecture: global → semantic → component.

Foundation (org-wide, locked) → Platform Extension (product-area-specific, reviewed) → LOB Extension (team-specific, self-governed in Explore)

Comprehensive Risk Analysis

Five risk categories with specific mitigations. Likelihood and impact assessed honestly.

Adoption Risks

Designers don't adopt — identity shift too threatening

High Likelihood High Impact

Not a training problem. "Designer" has meant "person who creates visual artifacts in Figma" for a decade. Code-first changes what designer means. That's an identity shift.

  • Phase 1 introduces Claude as thinking partner before changing creation workflow
  • Update evaluation criteria proactively: judgment quality, articulation, skill authoring, collaboration
  • Create space for resistance without judgment — "this feels threatening" is legitimate
  • Elevate early adopters as proof points; peers succeeding is more convincing than mandates

Engineering ignores prototypes and re-implements from scratch

Medium-High High Impact

Eliminates the handoff speed gain entirely.

  • Co-author Foundation skill with engineering — if they see their standards, they trust the output
  • Position as "living design proposals, not production code"
  • Joint pilot: engineer works with designer's prototype on a real feature

Shadow AI persists

High Likelihood Medium Impact

High if the org path feels slower. Ungoverned data exposure.

  • Make the org path genuinely better, not just more governed
  • Don't over-restrict Explore Mode — creative freedom with a quality floor
  • If detected, treat as a product problem first, not a compliance problem

Quality Risks

Automated checks create false confidence

Medium

Axe-core catches ~30-40% of accessibility issues. The machine-detectable ones. Not confusing tab order or misleading focus management.

  • Document explicitly: "Automated checks are a floor, not a ceiling"
  • Human sign-off is not optional even when all automated checks pass
  • Periodic manual accessibility audits compare findings to automated coverage

Output passes every check but fails users

Medium

Technically compliant but confusing interaction patterns. Automated checks can't catch this.

  • This is why challenge-hypothesis and user testing exist
  • Automated checks validate compliance; user testing validates utility
  • Faster building means more time available for testing — if teams don't skip it

Platform & Vendor Risks

Pricing Changes

Track per-team consumption monthly. Set usage caps. SKILL.md format is portable to Cursor, Codex, Gemini CLI. You're not locked in.

Model Behavior Changes

Test suite is the regression detector. Run after known model updates. Subscribe to Anthropic's changelog. Document workarounds immediately.

Key Person Leaves

Every skill has owner + deputy. Skills must include a "Design Rationale" section explaining why rules exist. Ownership transfer is explicit before departure.

Governance Becomes Obsolete

Invest heavily in design knowledge (durable). Invest lightly in governance tooling (volatile). A spreadsheet for the registry is fine.

Cultural & Organizational Risks

Workforce Implications

If designers generate working prototypes, what happens to junior front-end roles focused on translating mockups? Don't pretend it won't change the workforce. Commit to assessing implications at the 6-month milestone. Monitor role satisfaction and workload quarterly.

Is the Strategy Worth It If Claude's Design Output Doesn't Materially Improve?

Yes — but only if expectations were set correctly. The value is in automated compliance, handoff elimination, faster variant testing, and reusable encoded knowledge. NOT in replacing designer judgment on visual quality.

Phase 0: Governance Infrastructure (Weeks 1-2)

  • Assign central team ownership (Foundation skill author, governance lead)
  • Create central skills repository (git)
  • Create Approved Skill Registry (spreadsheet)
  • Configure Enterprise plan (managed settings, compliance API)
  • Interview 3-4 PMs and engineering leads for cross-functional temperature check

Phase 1: Foundation + Starter Kit (Weeks 2-4)

  • Author Foundation skill (co-authored with engineering)
  • Author CLAUDE.md template and starter kit skills
  • Build test suite (10-15 scenarios)
  • Build first-week onboarding module
  • Update designer evaluation rubrics

Phase 2: Pilot (Weeks 4-8)

  • Recruit one scrappy LOB team (Explore Mode) and one enterprise LOB team (Ship Mode)
  • Include one PM per pilot team
  • Teams begin Phase 1 of practice transition
  • PM tests read-filter-prd on a real PRD — debrief on usefulness
  • Collect weekly feedback; iterate Foundation skill (expect 2-3 versions)

Phase 3: Expand (Weeks 8-12)

  • Roll out starter kit to all willing teams
  • Scrappy LOBs begin code-first for new features
  • Enterprise LOBs begin Ship Mode pipeline
  • Engineering teams adopt shared CLAUDE.md and Foundation skill
  • Set up CI/CD hooks; build validate-acceptance-criteria skill

Phase 4: Scale + Measure (Weeks 12-16)

  • All teams have starter kit installed
  • Enterprise/scrappy configuration guides published
  • Quarterly review cadence established
  • Metrics tracking begins

Phase 5: Mature (Months 4-6)

  • 6-month workforce assessment
  • Cross-LOB consistency review
  • Foundation skill major version
  • Evaluate advanced capabilities
